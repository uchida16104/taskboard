⚠️ Critical: Workflow Compliance

This Mermaid diagram is the single source of truth for TaskBoard workflow. All code implementations, GitHub Actions, and UI behaviors must strictly follow this flow.

Any deviation between this diagram and actual system behavior is considered a critical bug and must be fixed immediately.

Workflow Steps Explained

Step 1

💡 Idea

Someone has an idea for a task, feature, or improvement. This is the starting point of the workflow.

Step 2

📝 Create Discussion

The idea is posted as a GitHub Discussion in the "Ideas" category. Use AI prompts to help structure the discussion.

→ Create a new discussion
Step 3

🏷️ Label Selection (Decision Point)

A maintainer reviews the discussion and applies ONE of these labels:

✅ approved

Creates issue → Project board

❌ rejected

Closes discussion

🤔 needs-review

Keeps pending

Step 4a

📋 Open Task/Issue (if approved)

When labeled "approved", a GitHub Action automatically creates an Issue from the discussion. The issue links back to the original discussion.

Step 4b

🚫 Close Discussion (if rejected)

When labeled "rejected", a GitHub Action automatically closes the discussion. A comment explains the rejection.

Step 4c

⏸️ Pending (if needs-review)

When labeled "needs-review", no automatic action occurs. The discussion remains open for further evaluation.

Step 5

📊 Add to Project Board

The newly created issue is automatically added to the GitHub Project board. Initial status is set to "Backlog".

Steps 6-9

📦 → 📝 → ⚙️ → ✅ Board Progression

Tasks move through project board columns manually as work progresses:

Backlog → Todo → Doing → Done
